# Developer Instructions: Reusable Travel Discovery & Cultural Experience Reference (TravelPilot)

This document contains instructions, architectures, and design rules to be followed when building travel discovery and cultural experience projects. These guidelines are compiled based on the analyzed parameters of the reference repository.

---

## 0. PromptWars Compliance & Disqualification Prevention (CRITICAL)

To prevent disqualification during the hands-on functional evaluation, the project MUST adhere to the following rules:

### ⚠️ Will Get You Disqualified (Do NOT Do):
*   **Static / Hardcoded Pages**: Never show an outcome in the UI that is not produced by real backend logic or a working model.
*   **Mock or Fake Data**: Do not present sample or placeholder data as if it were real output.
*   **Hallucinated AI Responses**: All AI outputs in the UI must be generated by a real, working AI call, not static mock responses.
*   **False Positives**: Ensure all features that appear to work in a demo actually work under genuine evaluation and testing.

### ✅ Do This Instead (Requirements):
*   **Every feature you demo must run**: If a feature is visible in the project, it must actually work end-to-end.
*   **Real GenAI Calls**: When using Gemini/GenAI, wire up an actual, working API / model call (not canned/static responses).
*   **Dummy Login Details**: If using login / authentication, provide evaluators with clear dummy login credentials so they can access every feature.
*   **End-to-End Testing**: Test the entire application end-to-end to ensure there are no bugs or broken paths when evaluators walk through.

> [!IMPORTANT]
> **Bottom line**: Build features that genuinely work, rather than more features that don't. Prioritize end-to-end functionality and robustness over empty visual completeness.

## 1. Reference Directory Structure

Use this modular layout as a template when initializing new repositories:

```
├── backend/                    # FastAPI backend codebase
│   ├── travelpilot/            # Core package
│   │   ├── __init__.py
│   │   ├── adaptive.py         # Travel itinerary adaptation scoring logic
│   │   ├── api.py              # Routing, middleware, and CORS configuration
│   │   ├── auth.py             # Firebase verification & auth resolver
│   │   ├── config.py           # Configuration values mapping env vars
│   │   ├── fallback.py         # Fallback static generators
│   │   ├── gemini.py           # Gemini API integrations
│   │   ├── models.py           # Shared request/response contracts (Pydantic)
│   │   ├── repository.py       # Firestore data access layer
│   │   └── utils.py            # Date/Time & general helper utilities
│   ├── main.py                 # FastAPI application entrypoint
│   ├── requirements.txt
│   └── requirements-dev.txt
├── frontend/                   # Next.js UI codebase
│   ├── src/                    # Frontend source components and pages
│   ├── lib/                    # API client wrapper
│   └── package.json
└── tests/                      # Local tests directory
    ├── conftest.py
    └── test_api.py
```

## 2. Parameterized Design Rules (Ordered by Precedence)

### [HIGH PRECEDENCE] Parameter 1: Code Quality
* **Pydantic Validation**: All API contracts must be defined using Pydantic `v2` in `models.py`. Inputs must be verified before passing them to services.
* **Separation of Layers**: 
  * `api.py` handles route definitions and logging.
  * `service.py` orchestrates transactions.
  * `gemini.py` / `repository.py` act as external adapters.
* **Abstract Persistence**: Write repository patterns that allow swapping out data providers (e.g. switching between in-memory mock storage and production Firestore).
* **Format Checking**: Use automated formatters (`ruff`) to clean up styles and organize imports.

### [HIGH PRECEDENCE] Parameter 2: Problem Statement Alignment
* **Diagnostics Flow**: Implement diagnostic check questions to establish the user's initial baseline.
* **Rule-Based Adaptive Engine**: Choose learning pathways using evidence:
  * Incorrect answer + low confidence -> Review prerequisite concepts.
  * Incorrect answer + high confidence -> Provide similar practice with a different approach.
  * Correct answer + high confidence -> Increase challenge level.
* **Visible Adaptation Logic**: Maintain an engine reason text (e.g., *"Why this step: System lowered explanation level because you missed the previous exercise"*). Save this reason string to the database and display it in the user's interface dashboard.

### [MEDIUM PRECEDENCE] Parameter 3: Security
* **Auth Scoping**: Secure every endpoint using user verification dependencies. If the request context is marked `guest`, isolate their session using temporary workspace controls. If signed in, verify Firebase auth tokens.
* **Session Ownership**: Enforce strict session checks:
  ```python
  if session.owner_id != user.user_id:
      raise HTTPException(status_code=403, detail="Forbidden")
  ```
* **Strict Schema Bounds**: Apply `min_length`, `max_length`, and range bounds (`ge`/`le`) on client inputs. Verify session ID prefixes (e.g., matching pattern `^tp_[a-f0-9]{14}$`).
* **Secure Headers**: Restrict CORS to allowed origins only. Inject security headers on the server and client.

### [MEDIUM PRECEDENCE] Parameter 4: Efficiency
* **Fast LLM Engine**: Use `gemini-2.5-flash` for all dynamic teaching and diagnostic requests to minimize token latency.
* **Strict JSON Schemas**: Supply structured Pydantic schemas in the Gemini API `response_schema` configurations. Do not write manual custom parsers.
* **Deterministic Fallbacks**: Provide a `fallback.py` file with mock diagnostics and lesson cards. Use it as a runtime recovery path if API keys are missing or quota limit errors occur.
* **Keep Dependencies Lean**: Avoid heavy machine learning runtimes (e.g., PyTorch, runtime embeddings) on the API web container.

### [MEDIUM PRECEDENCE] Parameter 5: Google Services
* **Google GenAI SDK**: Utilize the modern `google-genai` client for all Gemini model content generations.
* **Firestore Persistence**: Save user states using Google Cloud Firestore in production configurations.
* **Cloud Run & Secret Manager**: Store server keys in GCP Secret Manager and mount them securely inside Cloud Run container settings.

### [LOW PRECEDENCE] Parameter 6: Testing
* **Test Isolation**: Ensure each test is isolated. Always clean data repositories during setup.
* **Mock LLM Endpoints**: Patch the Gemini client in testing suites (`conftest.py`) to return predetermined mock content without invoking the network.
* **Validate Edge Cases**: Write test coverage for missing keys, mismatched owner permissions, validation boundary violations, and network failures.

### [LOW PRECEDENCE] Parameter 7: Accessibility
* **No Color-Only Cues**: Never use color as the single method to present state changes (such as indicating correctness). Always pair with readable text labels.
* **Keyboard Focus outlines**: Ensure every clickable element is accessible and highlight focus outline classes clearly.
* **Dynamic Announcements**: Use `aria-live="polite"` on updates related to coaching status or engine diagnostics.
